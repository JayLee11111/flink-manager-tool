# Flink Manager 修改建议（轻量工具版：监控 + 维护）

> 参考了 Dinky 的信息组织思路，但本方案明确**不做开发平台**，仅聚焦：
> - Flink 集群与 Job 监控
> - Job 运维维护
> - 告警配置与治理
> - 简单、直接、好上手

---

## 1. 目标与边界

## 1.1 建设目标
将当前页面从“基础信息维护”升级为“轻量实用的 Flink 运维工具”，满足日常巡检与快速处理问题。

核心目标：
1. 快速看清运行状态（集群、Job、告警）
2. 快速执行常见维护动作（新增、修改、停启）
3. 保持页面简洁，降低学习与使用成本

## 1.2 明确不做
- 不做 SQL 编辑器
- 不做 Jar/SQL 开发发布流水线
- 不做多租户开发空间
- 不做复杂血缘分析
- 不做事件与审计系统

---

## 2. 信息架构建议（工具导向）

建议保持简单导航：
1. **总览（Dashboard）**
2. **集群（Clusters）**
3. **作业（Jobs）**
4. **告警（Alerts）**

说明：
- 当前已有 1~4，建议直接在现有页面增强，不额外扩展复杂模块。

---

## 3. 页面级修改建议

## 3.1 总览页（index.html）
当前：展示集群和 Job 列表。

建议增强：
1. **顶部状态卡片（轻量 KPI）**
   - 集群总数 / 在线集群数
   - Job 总数 / RUNNING / FAILED
   - 当前启用告警数
2. **异常优先区域**
   - 最近失败 Job（Top N）
   - 最近触发告警（Top N）
3. **快捷入口**
   - 一键跳到“新增集群 / 新增 Job / 新增告警”

价值：第一屏就能知道系统是否健康。

## 3.2 集群管理页（clusters.html）
当前：增删集群 + 打开 `/overview`。

建议增强：
1. 集群状态展示：在线/离线/未知
2. 最后检测时间（Last Check）
3. 集群标签（生产/预发/测试）
4. 手动连通性检测按钮

价值：从“地址列表”升级为“可用性列表”。

## 3.3 Job 管理页（jobs.html）
当前：支持手工添加 + 参数解析添加。

建议增强：
1. **运行态字段**
   - 当前状态、运行时长、最近 checkpoint 时间
2. **常用维护动作**
   - 启动、停止、重启（仅保留高频动作）
3. **筛选能力**
   - 按集群、状态、BU、job-type、关键字
4. **参数可视化**
   - 参数详情弹窗（JSON）
   - 高亮关键参数（jobname/database/bu/job-type）

价值：提高日常维护效率，减少人工排查时间。

## 3.4 告警页（alerts.html）
当前：增删 + 启停告警。

建议增强：
1. 告警级别（P1/P2/P3）
2. 告警规则条件（如连续失败次数、延迟阈值）
3. 抑制策略（静默窗口、重复合并）
4. 按集群/BU/Job 的通知路由

价值：告警更精准，减少噪音。

## 4. 数据模型建议（基于当前 localStorage 演进）

## 4.1 Job 模型建议字段
- `id`
- `clusterId` 或 `cluster`
- `status`
- `startTime`
- `uptimeSec`
- `checkpointTs`
- `restartCount`
- `params`
- `rawParams`
- `updatedAt`

## 4.2 Alert 模型建议字段
- `id`
- `severity`
- `rule`
- `cooldownSec`
- `enabled`
- `lastTriggeredAt`
- `lastResolvedAt`

---

## 5. 监控指标建议（先做最小闭环）

优先做最有用的 5 个：
1. Job 状态分布（RUNNING/FAILED/CANCELED）
2. Job 重启次数（近 1h/24h）
3. 最近 checkpoint 成功时间
4. 集群可达性（在线率）
5. 告警触发次数（近 24h）

这些指标足够支撑日常巡检与快速定位问题。

---

## 6. 告警策略建议（轻量实用）

建议默认规则：
1. Job FAILED 立即告警（P1）
2. 连续重启超过阈值告警（P1/P2）
3. checkpoint 超时告警（P2）
4. 集群不可达持续 N 分钟告警（P1）
5. 同类告警合并（如 5 分钟窗口）

建议保留：
- 维护时段静默
- 恢复通知（Resolved）

---

## 7. 演进路线（轻量优先）

## Phase 1（短期，马上可做）
- 总览页增加 KPI 卡片
- Job 页增加参数详情弹窗 + 筛选
- 告警页增加级别与规则条件

## Phase 2（中期）
- 接入真实 Flink REST API 获取运行态
- 集群连通性自动刷新
- 告警触发记录（简版）

## Phase 3（后期，可选）
- 批量运维动作（批量停启）
- 趋势图与容量预警

---

## 8. 对当前项目的落地顺序建议

1. 先做 **总览 KPI + Job 参数详情弹窗**（投入最小、收益最高）
2. 再补 **Job 筛选 + 告警级别/规则**
3. 最后接入真实后端数据（替换纯 localStorage）

目标是保持工具“轻、快、直接”，避免走向复杂平台化。

---

## 9. 结论

当前 Flink Manager 已有骨架，建议继续沿“轻量工具”方向演进：
- 不做开发平台
- 不做事件与审计
- 聚焦监控、维护、告警三件事

这样最符合“方便使用”的定位，也最容易快速落地并持续迭代。
