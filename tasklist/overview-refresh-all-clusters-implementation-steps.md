# 总览页全量刷新实现步骤

## 1. 梳理现状与改造点
- 定位总览页初始化入口（页面打开时会执行的加载逻辑）。
- 确认当前“集群列表获取”和“Job 列表获取”的数据访问函数。
- 明确总览页当前使用的数据结构与渲染入口，确保后续可直接覆盖更新。

验收：
- 输出一份明确的改造点列表（入口函数、数据函数、渲染函数）。

## 2. 增加“页面打开即触发刷新”
- 在总览页初始化流程中接入自动刷新逻辑。
- 保证每次进入总览页都会执行，不依赖手动操作。
- 避免重复绑定导致的重复请求（同一次打开仅触发一次）。

验收：
- 打开总览页后可观察到自动触发刷新请求。

## 3. 实现“遍历所有集群并发拉取 Job”
- 先请求集群列表。
- 基于集群列表并发请求各集群 Job 数据（推荐 Promise.allSettled）。
- 对成功结果统一聚合为总览所需结构。

验收：
- 能覆盖所有集群并完成数据聚合。
- 单个集群失败不影响其他集群返回。

## 4. 更新总览数据与渲染
- 用聚合后的最新数据替换总览页面旧数据。
- 确保关键统计与列表展示同步更新。
- 保证刷新完成后 UI 与数据一致。

验收：
- 页面展示与聚合结果一致。
- 再次进入总览页后可看到更新后的最新状态。

## 5. 增加刷新过程状态提示
- 刷新开始时展示“刷新中”状态。
- 刷新完成后关闭“刷新中”状态。
- 防止用户误判页面卡死。

验收：
- 刷新期间有清晰可见的进行中提示。

## 6. 增加异常提示与容错
- 部分集群失败：展示“部分刷新失败”提示并附失败集群信息。
- 全部集群失败：按需求选择“保留旧数据”或“显示空态”，并提示失败原因。
- 错误提示不应阻断后续再次刷新。

验收：
- 可稳定复现并验证部分失败、全部失败两类场景。

## 7. 回归与验收
- 场景1：正常场景，所有集群成功返回。
- 场景2：部分集群失败，其余成功。
- 场景3：全部失败。
- 场景4：连续多次进入总览页，均自动触发刷新。

验收：
- 满足 requirement 文档中的全部验收标准。
- 无明显性能退化（集群较多时仍可接受）。

## 8. 可选优化（不影响首版交付）
- 对失败集群支持重试按钮。
- 记录每次刷新耗时与失败率，便于后续优化。
- 增量更新局部区域，减少整页重绘开销。
